## Phase Scope Note

Phase 5 is predominantly a zero-UI SEO phase. The only visible user-facing change is a blog search text input (`?q=` parameter) added to the existing blog index page. The design contract is therefore minimal: it governs the single blog search input and its states.

## Blog Search Input — Interaction Contract

The blog search is the only interactive UI element added in this phase. It uses the existing `?category=` nuqs pattern.

### Input Field

- Component: `Input` from `src/components/ui/input.tsx`
- Height: `--size-input-height: 2.5rem` (40px)
- Leading icon: `Search` (lucide, 16px, `--icon-size-base`, `--color-muted-foreground`)
- Placeholder: "Search articles..." (13px, `--color-form-placeholder`)
- On focus: `box-shadow: --shadow-input-focus` (navy ring), border color to `--color-form-input-border-focus`
- Input type: `text` — do NOT use `type="search"` (browser native clear button conflicts with custom clear button)

### Clear Button

- Visible only when input value is non-empty
- Component: `Button` variant `ghost`, size `icon`
- Icon: `X` (lucide, 16px, `--color-muted-foreground`; on hover `--color-foreground`)
- Positioned absolutely at trailing edge of input container
- Touch target: `min-h-[44px] min-w-[44px]`
- Aria label: "Clear search"

### Results Behavior

- Filter is immediate (no submit button, no debounce needed at blog post count scale)
- Zero results: show empty state (see Copywriting Contract below)
- Filter applies to: post `title` and `excerpt` fields (case-insensitive includes match)
- URL state: `?q={value}` via `nuqs` `useQueryState` — shareable, browser-back safe
- Coexists with existing `?category=` filter (both apply simultaneously)

### Animation

- Input clear button fade-in: `opacity-0` → `opacity-100`, duration `--motion-duration-fast: 150ms`, ease `--motion-ease-out`
- No skeleton or loading state — filter is synchronous (client-side, posts already in memory)

## JSON-LD Components — Visual Contract

JSON-LD components produce no rendered HTML visible to users. They render only `<script type="application/ld+json">` tags in `<head>`. The following constraints apply:

| Constraint | Rule |
|------------|------|
| Script tag placement | Server-rendered at page load — no `'use client'` on any `json-ld/` component |
| Nonce prop | All components accept `{ nonce?: string | null }` and pass `nonce ?? undefined` to DOM |
| Serialization | Only `safeJsonLdStringify` from `src/lib/json-ld-utils.ts` — never raw `JSON.stringify` |
| URL values | Hardcoded `https://richardwhudsonjr.com` — never `siteConfig.url` (resolves to localhost in some envs) |

These are engineering constraints that affect correctness, not visual design.
